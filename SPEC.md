# СПЕКА — скіл `/statusline`

ШІ-збірка статус-бара Claude Code: дружня до новачків, портативна, з памʼяттю.
Статус: **спека узгоджується**. Коду ще нема. Будуємо через `write-a-skill` після
фінального узгодження.

> Документ ведеться українською (правила користувача). Code identifiers — англійською.

---

## 1. Призначення і філософія

ШІ веде будь-кого (навіть «зовсім не в курсі») через збірку статус-бара **розмовою**.
Не меню — діалог, де ШІ **пояснює, пропонує, питає по місцю**.

Наскрізний тон: *«вибір — це лише інформація. Він нічого не закриває. Завжди можеш
переграти, описати своє, або поправити руками».* Прогресивне розкриття: спершу простий
вибір, глибше — лише за бажанням.

Це «застосунок, згенерований ШІ для користувача»: сучасний ШІ-сетап, де клієнта питають
по місцю, а рішення лишаються оборотними.

---

## 2. Прецедент і що беремо з p10k

Найближчий доведений аналог — `p10k configure` (powerlevel10k, zsh): інтерактивний
майстер, що питаннями збирає prompt. Тримається ~5 років. Беремо свідомо:

1. **Визначати середовище очима користувача.** Скрипт не бачить ширини/шрифту терміналу
   (перевірено: нема поля в JSON, нема tty, `/dev/tty` недоступний). Рішення p10k —
   спитати людину що вона *бачить*. → robимо glyph-check: показати тестовий рядок з
   `█ ▒ ░ | ⏷ Δ`, спитати «всі символи видно чи квадратики/знаки питання?» → якщо ні,
   падаємо на ASCII-бари (`#`/`-`, `>` замість `|`, без `⏷/Δ`-гліфів).
2. **Розділити декларативний конфіг і рушій.** `~/.p10k.zsh` — змінні з коментарями,
   рушій фіксований. Через це не старіє. → **наша архітектура §3**.
3. **Конфіг сам себе документує.** Кожна опція з коментарем-поясненням → маніфест з
   коментарями біля кожного показника; файл навчає і запрошує правити руками.
4. **Воронка рішень + «(recommended)».** Велике → деталі, скрізь дефолт.
5. **Неруйнівність + ідемпотентність.** Бекап старого, перезапуск будь-коли.

Чого p10k структурно не може і що є нашою новизною: **ШІ пояснює коли показник — шум**.

## 3. Архітектура: стабільний рушій + декларативні профілі

Рішення (під впливом p10k, розвертає попередню думку «генерувати скрипт під профіль»):

- **Рушій** — один стабільний `statusline.sh`, шиплений зі скілом. Читає активний
  маніфест щоразу при рендері й малює. Пишеться раз, далі стабільний.
- **Профіль** — декларативний **маніфест** (які показники, ширини, пороги, коментарі).
- **Перемикання профілю** = змінити вказівник `active`, без регенерації.
- **Старий бар** користувача імпортується як перший профіль `previous` (так «зберегти
  старе» — частковий випадок профілів, одна механіка).
- ШІ редагує **чистий маніфест**, не переписує bash щоразу → надійніше й безпечніше.

Переваги: профілі, збереження старого, і майбутня автоматизація — все випадає з однієї
механіки. Ціна: рушій складніший (умовний рендер усього), але пишеться один раз.

### Структура файлів (чернетка)
```
~/.claude/statusline/
  engine.sh                 # стабільний рушій (рендерить активний маніфест)
  active                    # імʼя активного профілю
  profiles/
    minimal.conf
    balanced.conf
    full.conf
    previous.conf           # імпортований старий бар користувача
  state.json                # памʼять: історія, відкинуте, журнал тертя
  backups/
    statusline.sh.<ts>      # сирий бекап того що було до скіла
```
`settings.json → statusLine.command` вказує на `~/.claude/statusline/engine.sh`.

## 4. Каталог показників (серце продукту)

Для кожного — *що / навіщо / коли шум*, простою мовою (текст для діалогу ШІ):

| показник | що це | навіщо | коли шум |
|---|---|---|---|
| **ctx бар+токени** `.11/1` | скільки контексту зʼїдено / вікно | бачити близькість до стиснення | майже ніколи — база |
| **ctx %** | те саме у % | — | дублює бар → дефолт off |
| **вартість $** | гроші за сесію | стежити за витратами | на безлімітному плані байдуже |
| **ліміт 5h бар+%** | скільки квоти плану зʼїв за 5 год | не впертись у стелю | якщо завжди далеко від межі |
| **ліміт 7d бар+%** | те саме за 7 днів | довша перспектива | новачку часто зайве |
| **маркер темпу `|`** | де лінійно мав би бути за часом | миттєво: випереджаєш чи ні | без пояснення незрозуміло |
| **зона-прогноз `▒/▓`** | куди прийдеш у цьому темпі | рання попередження | складно новачку |
| **Δ час заблокований** | скільки чекатимеш як виб'єш межу | найдієвіше — каже наслідок | якщо рідко впираєшся |
| **час скидання `⏷`** | коли вікно оновиться | знати скільки терпіти | — |
| **модель** | яка модель активна | підтвердження | рідко змінюється |
| **ефорт** | рівень зусиль | фіча користувача | якщо не користуєшся |

Дані всі з JSON payload (`context_window`, `rate_limits.{five_hour,seven_day}`, `cost`,
`model`). **`cs` та зовнішні залежності — викинуті.** Єдина залежність — `python3`.

## 5. Пресети (розвʼязують «складно лякає новачка»)

- **Мінімум** — `ctx бар+токени · модель`. Чистий, нестрашний.
- **Збалансований** *(дефолт новачку)* — `ctx · 5h бар+%+⏷ · модель`. Без прогнозу/Δ/7d.
- **Все** *(поточний у Олекси)* — `ctx · 5h+7d з маркером+зоною+Δ · модель+ефорт`.

Новачок тицяє пресет і все. Хто хоче — заглиблюється в перемикачі, кожен з поясненням §4.

---

## 6. Памʼять і журнал тертя («memorful»)

Механізм уже є: скіл володіє `state.json` + у Claude Code власна памʼять. Памʼятає:
- активний профіль;
- що користувач раніше **обрав** і що **відкинув** (щоб не перепитувати);
- журнал тертя: що міняли руками після генерації, що плутало.

Наступний запуск: *«минулого разу прибрав вартість — лишаємо?»*

**Самовдосконалення — свідоме, не автономне.** Скіл НЕ переписує сам себе (крихко, важко
ревʼюити). Він **збирає тертя у журнал**, а покращуємо ми вручну, за потреби з евалами
`skill-creator`. Памʼять + її використання дає відчуття «він вчиться» без саморуйнації.

---

## 7. Портативність і деградація

- Нема `python3` → чесно кажемо «треба python3», пропонуємо як поставити, не падаємо.
- Гліфи не рендеряться (glyph-check §2 провалився) → ASCII-режим маніфесту.
- Поля JSON відсутні (стара версія Claude Code) → ховаємо секцію, не ламаємось.
- Жодних жорстких шляхів, жодного `cs`, жодної macOS-специфіки.

---

## 8. Потік розмови

1. **Тихо визначити середовище** — є `python3`? яка ОС? чи є вже `statusline` (→ бекап).
2. **Glyph-check** — показати тестовий рядок, спитати чи все видно (визначає ASCII-fallback).
3. **Привітати + 2 рядки** що таке статус-бар і що вибір оборотний.
4. **3 пресети** з живим прев'ю кожного (рендер реальним кодом).
5. *(опційно)* заглиблення в перемикачі — дефолт від пресета, кожен з поясненням.
6. **Фінальний рендер** → підтвердження.
7. **Бекап** старого (як профіль `previous`) → запис маніфесту → вказівник у `settings.json`.
8. **Сказати як міняти потім**: перезапустити `/statusline`, перемкнути профіль, або руки.

---

## 9. Формат маніфесту (чернетка, з дверима під автоматизацію)

Декларативно, з коментарями (дух p10k). Парситься рушієм (python).
Рядок: `показник = on/off [параметри]  # пояснення`.

```
# profiles/full.conf — приклад
ctx        = on   width=12 tokens=2dec        # контекст: бар + .11/1
ctx_pct    = off                              # % контексту (дублює бар)
cost       = off                              # вартість сесії
limit_5h   = on   width=12 marker projection delta
limit_7d   = on   width=12 marker projection delta
reset_time = on                               # ⏷ час скидання
model      = on
effort     = on
glyphs     = unicode                          # або ascii (з glyph-check)

# --- rules: ПОРОЖНЬО у v1, рушій ігнорує. Двері для автоматизації v2 ---
# rule: dir=~/Projects/Nominal  -> profile=work
# rule: model=haiku             -> profile=minimal
```

**Рішення (узгоджено):** формат закладаємо під майбутні `rules`, але **рушій правил не
пишемо у v1**. Двері лишаємо — кімнату не будуємо.

---

## 10. Вивід, бекап, безпека

- Перед будь-яким записом — бекап існуючого `statusline.sh` у `backups/` + імпорт як
  профіль `previous`.
- Запис атомарний; якщо щось не так — лишається старе.
- Нічого поза `~/.claude/statusline/` та рядком `statusLine` у `settings.json` не чіпаємо.

---

## 11. Межі v1 / що у v2

**v1:** статус-бар; рушій+маніфести; 3 пресети; каталог пояснень; glyph-check;
портативність (нуль зовнішніх залежностей крім python3); профілі; памʼять+журнал тертя;
бекап старого як профіль.

**НЕ v1:** рушій правил автоматизації; ширший сетап Claude Code (skills/permissions/
CLAUDE.md); автономне самопереписування скіла.

**v2 кандидати:** автоматизація профілів за контекстом (читати `current_dir`/`model` з
JSON); евал-цикл покращення через `skill-creator`; розширення на інші частини сетапу.
